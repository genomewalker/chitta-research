What makes it different

Most auto-research systems forget everything between runs and have no notion of epistemic confidence. chitta-research accumulates structured knowledge.

⬡

Typed belief graph, not a log file

Eight node types connected by five epistemic edge kinds. Every hypothesis carries prior and posterior confidence. Every claim records which observations support it. The graph serializes to JSON and restores on the next run — nothing is discarded.

◈

6-dimensional Pareto fitness

Experiments scored on novelty, empirical gain, reproducibility, cost efficiency, transfer potential, and calibration improvement. The Pareto frontier is maintained continuously. No single weighting scheme is imposed.

↯

Reconsolidation propagates corrections

When a high-priority belief is corrected, confidence reductions cascade immediately to all downstream nodes via DerivedFrom traversal. BlockedBy warning edges are inserted in the same cycle — not retrospectively.

◉

chitta-field native

Each agent writes memories, triplets, and observations to the chittad daemon. Failed experiment outcomes lower domain recall scores so future hypothesis generation is aware of dead ends. Persistent across research sessions.

⊞

Multi-model debate rooms

The room provider runs a structured parallel debate across Claude, local Ollama models (gemma4, deepseek-r1), and Codex before producing a synthesized structured answer. Mix backends per participant. Brahman monitors convergence and stores cross-domain insights in chitta memory.

⌘

No API key required by default

The claude-cli provider delegates all LLM calls to claude -p, reusing the authenticated Claude Code session. No separate billing configuration. Local models via Ollama need no credentials at all.

⊕

Git-backed artifact store

Every experiment result is committed to a local git repo. Commit SHAs are stored on Run nodes. Full provenance chain: graph node → git commit → file content. Nothing is lost.

Architecture

Three Tokio tasks run concurrently, each polling its own step() on a 100 ms loop against a shared belief graph.

The orchestrator in cr-daemon shuts down when all agents are idle for --max-cycles consecutive cycles, the budget is exhausted, or SIGINT is received. The final graph snapshot is written to graph_state.json.

Belief graph

A petgraph-backed stable directed graph with typed lookup, evidence queries, and JSON snapshot I/O. The graph accumulates across sessions — it is never reset between runs.

Node types

ResearchProgram Question Hypothesis ExperimentPlan Run Observation Claim Method

Edge kinds

Supports Refutes DerivedFrom GeneralizesTo BlockedBy

Hypothesis node

Carries prior and posterior confidence (0.0–1.0). Prior is set by Hotr at generation. Posterior is updated by Udgatr after scoring. Reconsolidation reduces it when upstream beliefs are corrected.

Run node

Stores status, timestamps, token + cost accounting, and a git commit SHA pointing to the committed results.json artifact. Udgatr adds a FitnessVector once the run succeeds.

The four agents

Each agent implements the Agent trait. They run concurrently in separate Tokio tasks, polling step() and writing to the shared belief graph and chitta daemon.

Hotr
Hypothesis generation

Reads the belief graph, finds the highest-priority ResearchProgram, then finds its unanswered Question nodes — those with no DerivedFrom children of type Hypothesis. Sends the question and program context to the LLM. Each returned hypothesis is added as a Hypothesis node with an ExperimentPlan child. Prior confidence is set by the LLM.

Adhvaryu
Experiment executor

Finds ExperimentPlan nodes with no Run child. Sends the plan steps and hypothesis statement to the LLM. A Run node is added with status, timestamps, token/cost accounting, and a git commit SHA pointing to the committed artifact. Observation nodes are added for each observation string. Writes hypothesis generation and experiment outcome events to chitta.

Udgatr
Analyst & scorer

Finds Run nodes with status = Succeeded and no fitness score. Sends hypothesis, plan, and observations to the LLM for structured scoring across all six fitness dimensions. Updates the run's FitnessVector and the hypothesis posterior confidence. For results with empirical_gain > 0.6, adds Claim nodes with Supports or Refutes edges. Writes a triplet to chitta recording the verdict.

Brahman
Portfolio arbiter

Monitors convergence across all participants in a multi-model room. Detects stagnation, scores novelty of each round, and stores cross-domain transfer opportunities in chitta memory so future sessions can draw on insights from adjacent research programs. The arbiter that decides when the debate has yielded enough signal.

6-dimensional Pareto fitness

No single weighting scheme is imposed. The Pareto frontier over all scored runs is maintained continuously. Dominated results are pruned without losing the trade-off structure.

Novelty
How different is this result from existing knowledge in the belief graph?
Empirical gain
How much does this result improve confidence in the target hypothesis?
Reproducibility
How likely is this result to replicate under independent conditions?
Cost efficiency
Information gain per dollar of token spend and compute used.
Transfer potential
How broadly applicable is this result beyond the immediate question?
Calibration improvement
How much does this result sharpen the model's own epistemic confidence?

Agenda format

The agenda is the entry point. It declares research programs, budget, LLM backend, and chitta mind path. Each question becomes a Question node; each method becomes a Method node.

programs:
  - title: "ARG prevalence in ancient permafrost"
    domain: metagenomics
    questions:
      - "Do antibiotic resistance genes increase with permafrost depth?"
      - "Are ancient ARGs phylogenetically distinct from modern variants?"
    methods:
      - "BLASTp against CARD"
      - "Short-read assembly with MEGAHIT"
    priority: 1.0
    max_budget_usd: 5.0

budget:
  total_usd: 10.0
  gpu_slots: 0
  cpu_workers: 4

llm:
  provider: claude-cli        # no API key required
  model: claude-sonnet-4-6

chitta:
  mind_path: ~/.claude/mind

LLM providers

All providers implement exponential back-off with 3 retries on 429/503. The default claude-cli backend needs no configuration beyond an active Claude Code session.

Provider Key in YAML What it does
claude-cli default Spawns claude -p --output-format text, reads stdout. No API key.
anthropic anthropic Direct HTTP to api.anthropic.com. Requires ANTHROPIC_API_KEY.
openai openai OpenAI-compatible HTTP. Requires OPENAI_API_KEY.
codex codex Spawns codex exec -. Model set via model: field.
chitta-bridge chitta-bridge Routes through chitta-bridge --exec. Supports opencode, codex, and local backends from a single interface with session management.
openai-compat openai-compat + base_url Direct HTTP to any Ollama or vLLM /v1 endpoint. No credentials required. Use for local models (gemma4, deepseek-r1, etc.).
room room Multi-participant parallel debate. Each participant uses a different backend and model. Synthesizer produces the final structured answer. See below.
mock --mock flag Deterministic stub. Full pipeline without any external process. For testing.

Multi-model rooms

The room provider runs a structured multi-participant debate before producing a synthesized answer. Each participant uses a different backend and model. All participants run in parallel per round; the synthesizer runs once at the end.

llm:
  provider: room
  model: "aDNA compression codec selection"
  room:
    rounds: 1
    participants:
      - name: Critic
        backend: claude-cli
        system: "You are a bioinformatics critic..."

      - name: Empiricist
        backend: openai-compat
        model: gemma4:26b
        base_url: http://gpu-node:11434/v1
        system: "You are an empiricist..."

      - name: Theorist
        backend: openai-compat
        model: deepseek-r1:32b
        base_url: http://gpu-node:11434/v1
        system: "You are an information theorist..."

      - name: Engineer
        backend: codex
        model: gpt-5.4
        system: "You are a systems engineer..."

    synthesizer:
      name: Synthesizer
      backend: claude-cli
      system: "You are a JSON synthesizer. Resolve any contradictions
               between participants, then produce the final structured
               answer. Output raw JSON only — no prose, no markdown."

See agenda.test-local.yaml in the repo for a complete working example — four models debating aDNA FASTQ compression codec selection, with Brahman monitoring convergence and storing cross-domain transfer opportunities in chitta memory.

chitta integration

The cr-chitta crate opens a Unix socket connection to chittad and speaks JSON-RPC using the MCP tools/call wire format. Three operations carry the research memory layer.

✦

remember

Hotr records hypothesis generation events. Adhvaryu records experiment outcomes with tags [adhvaryu, experiment, succeeded|failed]. Failed outcomes lower domain recall scores — future hypothesis generation is aware of dead ends.

✦

observe

Reconsolidation writes structured observations under the reconsolidation category so the daemon's subconscious can process them during its next consolidation cycle.

✦

connect (triplet)

Udgatr writes hypothesis:ID confirmed_by run:ID (or refuted_by) with the posterior confidence as weight. The knowledge graph accumulates research verdicts across all sessions.

The daemon is optional. If the socket is unavailable, ChittaClient::connect() fails and a warning is logged — all agent logic continues without it. chitta is an enhancement, not a hard dependency.
